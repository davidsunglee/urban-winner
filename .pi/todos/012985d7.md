Implement the common eval harness that executes each framework implementation against shared benchmark use cases through the repository contract.

## Goal

Build the framework-agnostic harness in `evals/` so it can discover framework manifests, prepare per-run inputs, invoke each agent through `shared/contract.md`, collect outputs/traces, score results against `shared/task-spec.md`, and produce a comparative report.

## Scope

- Discover framework implementations under `frameworks/` via per-framework `manifest.json` files or equivalent entrypoint metadata.
- Invoke each framework without importing framework-specific code, using the subprocess JSON stdin/stdout contract by default.
- Load one or more benchmark cases from shared fixtures/use-case definitions.
- For the software bugfix benchmark, create an isolated mutable worktree per `(framework, case, run)` from the canonical fixture repository.
- Pass the task-specific `input` shape defined in `shared/task-spec.md`, including `case_id`, `repo_path`, `failing_test_command`, `failure_output`, and `edit_constraints`.
- Enforce harness-owned responsibilities: worktree lifecycle, diff derivation, visible test rerun, optional hidden test rerun, edit-constraint compliance, trace capture, scoring, and cleanup.
- Validate response envelope and task output schema.
- Report independent scoring categories rather than a single weighted aggregate.
- Generate a human-readable comparative report across frameworks/cases/runs.

## Candidate modules

- `evals/__main__.py` — CLI entrypoint, e.g. `run --all`, `run --framework <name>`, `run --case <id>`.
- `evals/runner.py` — manifest discovery, subprocess invocation, timeout/error handling.
- `evals/cases.py` — case loading and validation.
- `evals/worktrees.py` — fixture worktree creation/reset/cleanup.
- `evals/scoring.py` — schema validity, test outcomes, edit constraints, minimality, trace quality, latency/token extraction.
- `evals/report.py` — comparative table/report generation.

## Acceptance criteria

- `just eval-all` can discover all configured framework manifests and attempt to run them through the shared contract.
- `just eval <framework>` can run a single framework.
- The harness can run at least one software bugfix fixture case end-to-end once a framework implementation exists.
- Each run stores enough raw artifacts to debug failures: request JSON, response JSON or error JSON, stderr/logs, harness-derived diff, visible/hidden test output, and score JSON.
- Scoring output includes separate categories: schema validity, visible test outcome, hidden test outcome when applicable, edit-constraint compliance, minimality, trace quality, latency, and token usage when available.
- Worktrees are isolated per framework/case/run and are cleaned up or retained according to an explicit debug/keep-artifacts option.
- The harness does not import framework implementation code directly; all framework-specific behavior stays behind manifests/adapters.

## Notes

This todo is about the common harness only, not implementing individual framework agents. It should build on the v1 software bugfix benchmark contract in `shared/task-spec.md` and leave room for future data incident and review/planning benchmark use cases.
